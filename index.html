Hi! My name's Nina.

And I made a Minecraft datapack for my senior thesis.

Contrary to what my parents think, I haven't just been playing Minecraft for the last few months. I've been modifying it.

I wanted to tap in to the basic human tendency to look up at the stars. In a Minecraft world, which so clearly parallels the human experience, stars dot the night sky. But players have wondered for years: What do they mean?

Surprisingly, all stars in Minecraft are generated from the same seed. Meaning: all stars in every Minecraft world are generated from the same random number.

This got me thinking not only about my place in this universe, but the place of every Minecraft world and every Steve and Alex in every Minecraft world. What stories would the stars tell?

And so, I got to work making a datapack all about those pixels in the night sky.

What's a datapack?

A Minecraft datapack is a bunch of code that does stuff in Minecraft. For those familiar with Command Blocks, a datapack is just some files that hold a bunch of command block commands. Datapacks can also do more to the game than command blocks can, adding special advancements, loot tables, recipes, and structures, among other things. Datapack functions, the files that hold the command block lines of code that actually execute while playing Minecraft, are .mcfunction files. The coding language is identical to the command block code, and in fact reddit.com/r/MinecraftCommands has been immensely helpful in my project.

Each .mcfunction file is its own method, to use standard coding jargon, but methods can not take in any parameters, and each line of code is indepdenent from the other lines (basically, there are no multi-line code statements like if-statements with brackets, for-loops, while-loops, etc.). Instead, to share information between lines of code and functions, you have to use Scoreboards. Scoreboards are basically a more convoluted version of your standard variable. Instead of being able to simply declare:

var a = 0;

you have to first set up a Scoreboard Objective within which to hold Socreboard players, which act like variables. Like this:

scoreboard objectives add testing dummy

scoreboard players add a testing 0

There are more minutia about how Scoreboards work and their seemingly infinite amount of restrictions, but that's a basic overview. Also notably, you can't use scoreboard values dynamically. For example, you can't teleport directly to a set of scoreboard values, but would have to write a specific function that takes in a scoreboard value for the x, another value for the y, and another value for the z. The function might look something like this:

execute if testing dummy < 15000 constant run tp PlayerName 7500 ~ ~

execute unless testing dummy < 15000 constant run tp PlayerName 22500 ~ ~ ~

Etc. etc., teleporting the player half and then half again until reaching the exact point in the Scoreboard values, instead of being able to read them and just teleport the player right there. These are just some of the many restrictions datapacks have, and for good reason.

My impression is that datapacks were deisgned to be just like command blocks, which themselves are designed for new coders. Instead of complex ideas like primitive data types and arrays, command blocks simply deal with the realities of Minecraft, which are mainly numbers. Thus, Scoreboards can only hold integers, though sometimes there are functions in the game that ask for a "scale", meaning Scoreboard theoretically hold doubles and floats. But again, no strings at all.

This all may seem incredibly restrictive, but often that's what the most creative solutions need: barriers, challenges, and restirctions.

Why a datapack?

This project originally started as a Minecraft modification (or "mod"), which basically means "actual" coding in a language like Java (see my process for more information on what this project almost was). But I eventually settled on a datapack because of its simplicity. It doesn't promise any kind of complex API, but instead promises a list of simple commands you can run in Minecraft to achieve the seemingly impossible. There are a whole range of incredible things you can do with datapacks, including adding seasons, furniture, and even space adventures.

The other advantage of a datapack is that it's incredible easy to install and once the datapack is in your datapacks folder, that's it. They all work multiplayer by design (since they're just command block codes there's also no need for anything special server-side: just put the datapack on the server map and that's it, everyone's using the datapack!). There's also a truly thriving community of datapack creators since it's much easier to get into datapacking than modding (since modding requires knowing a complex, object-oriented language, as well as altering the Minecraft client).

Datapacks are also very easy to test-- instead of recompiling all of Minecraft like you would to make a client-side mod like a Forge mod, since datapacks are built in to the experience of Minecraft you can code a datapack, save the file (and even keep it open on your editor) and simply type /reload to test out your newest code. That's it! You can even refresh textures while you're playing Minecraft with T + F3. This way you can see the code you've built immeditately, and more easily make changes and tweaks necessary while the game is running.
